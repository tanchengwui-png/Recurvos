RECURVOS DEPLOYMENT CHEAT SHEET
================================

Branch rules
- `develop` = staging branch
- `main` = production branch
- Never work directly on `main`
- Never deploy a feature branch directly to production

Server paths
- Staging repo path: `/srv/recurvos/staging/repo`
- Production repo path: `/srv/recurvos/prod/repo`

Production branch rule
- Production server must run `main`

Recommended domain env vars

Production
```text
APP_WEB_BASE_URL=https://app.recurvos.com
APP_API_BASE_URL=https://api.recurvos.com
VITE_API_BASE_URL=https://api.recurvos.com/api
VITE_PUBLIC_SITE_URL=https://recurvos.com
VITE_APP_SITE_URL=https://app.recurvos.com
```

Staging
```text
APP_WEB_BASE_URL=https://staging-app.recurvos.com
APP_API_BASE_URL=https://staging-api.recurvos.com
VITE_API_BASE_URL=https://staging-api.recurvos.com/api
VITE_PUBLIC_SITE_URL=https://staging.recurvos.com
VITE_APP_SITE_URL=https://staging-app.recurvos.com
```

Local development to staging to production
-----------------------------------------

1. Start your work from `develop`
```powershell
cd C:\recurvos
git checkout develop
git pull origin develop
git checkout -b feature/short-description
```

2. Make your code changes locally

3. Verify locally before merging
```powershell
cd C:\recurvos\src\Recurvos.Web
npm.cmd run build

cd C:\recurvos
dotnet build src\Recurvos.Infrastructure\Recurvos.Infrastructure.csproj
```

4. Commit and push your feature branch
```powershell
cd C:\recurvos
git status
git add .
git commit -m "Short clear message"
git push -u origin feature/short-description
```

5. Merge your feature branch into `develop`
```powershell
cd C:\recurvos
git checkout develop
git pull origin develop
git merge feature/short-description
git push origin develop
```

6. Deploy `develop` to staging
```bash
ssh server@115.187.22.70 -p 6262
cd /srv/recurvos/staging/repo
git status
git branch --show-current
git log --oneline -5
git checkout develop
git fetch origin
git log --oneline origin/develop -5
git pull origin develop
docker compose -f docker-compose.staging.yml up -d --build api web
```

7. Verify staging
```bash
cd /srv/recurvos/staging/repo
docker compose -f docker-compose.staging.yml ps
docker logs --tail 100 recurvos-staging-api
docker logs --tail 100 recurvos-staging-web
docker logs --tail 100 recurvos-staging-worker
```

Staging update when `git pull` is blocked by local changes
----------------------------------------------------------

If staging says a file is modified and refuses to pull, do this first.

1. Check what changed
```bash
cd /srv/recurvos/staging/repo
git status
git diff -- docker-compose.staging.yml
```

2. If the change is a server-only change you want to keep, stash it
```bash
git stash push -m "staging local change" -- docker-compose.staging.yml
```

3. Pull the latest `develop`
```bash
git fetch origin
git checkout develop
git pull origin develop
git log --oneline -5
```

4. Re-apply the stash only if you still need it
```bash
git stash list
git stash pop
```

5. Rebuild the containers you actually changed
```bash
docker compose -f docker-compose.staging.yml up -d --build api web
```

6. Verify the running code
```bash
docker compose -f docker-compose.staging.yml ps
docker logs --tail 100 recurvos-staging-api
docker logs --tail 100 recurvos-staging-web
```

Expected result
- `git branch --show-current` should be `develop`
- `git log --oneline -5` should show the latest commit from `origin/develop`
- rebuild `api` whenever backend code or EF migrations changed
- rebuild `web` whenever frontend code changed
- rebuild both `api web` when a feature touches subscription behavior end to end

8. If staging is good, merge `develop` into `main`
```powershell
cd C:\recurvos
git checkout main
git pull origin main
git merge develop
git push origin main
```

9. Deploy `main` to production
```bash
ssh server@YOUR_PROD_IP -p YOUR_PORT
cd /srv/recurvos/prod/repo
git checkout main
git pull origin main
docker compose -f docker-compose.yml up -d --build postgres mailpit api web worker
```

10. Verify production
```bash
cd /srv/recurvos/prod/repo
docker compose -f docker-compose.yml ps
docker logs --tail 100 recurvos-prod-api
docker logs --tail 100 recurvos-prod-web
docker logs --tail 100 recurvos-prod-worker
```

Fast deploy options
-------------------

If you only changed backend code on staging
```bash
cd /srv/recurvos/staging/repo
docker compose -f docker-compose.staging.yml up -d --build api worker
```

If you only changed frontend code on staging
```bash
cd /srv/recurvos/staging/repo
docker compose -f docker-compose.staging.yml up -d --build web
```

If you only changed backend code on production
```bash
cd /srv/recurvos/prod/repo
docker compose -f docker-compose.yml up -d --build api worker
```

If you only changed frontend code on production
```bash
cd /srv/recurvos/prod/repo
docker compose -f docker-compose.yml up -d --build web
```

Hotfix flow
-----------

1. Start from `main`
```powershell
cd C:\recurvos
git checkout main
git pull origin main
git checkout -b hotfix/short-description
```

2. Make the hotfix and verify locally

3. Commit and push the hotfix
```powershell
cd C:\recurvos
git add .
git commit -m "Hotfix short clear message"
git push -u origin hotfix/short-description
```

4. Merge hotfix into `main`
```powershell
cd C:\recurvos
git checkout main
git pull origin main
git merge hotfix/short-description
git push origin main
```

5. Deploy production from `main`
```bash
ssh server@YOUR_PROD_IP -p YOUR_PORT
cd /srv/recurvos/prod/repo
git checkout main
git pull origin main
docker compose -f docker-compose.yml up -d --build postgres mailpit api web worker
```

6. Back-merge the hotfix into `develop`
```powershell
cd C:\recurvos
git checkout develop
git pull origin develop
git merge main
git push origin develop
```

7. Redeploy staging so `develop` matches production
```bash
ssh server@115.187.22.70 -p 6262
cd /srv/recurvos/staging/repo
git checkout develop
git pull origin develop
docker compose -f docker-compose.staging.yml up -d --build postgres mailpit api web worker
```

Bugfix to staging
-----------------

Use this when you already fixed the bug locally and just want to push and refresh staging.

1. Commit and push the fix to `develop`
```powershell
cd C:\recurvos
git checkout develop
git status
git add .
git commit -m "Short bugfix message"
git push origin develop
```

2. Update the staging repo
```bash
ssh server@115.187.22.70 -p 6262
cd /srv/recurvos/staging/repo
git status
git fetch origin
git checkout develop
git pull origin develop
git log --oneline -5
```

3. Rebuild only the services affected by the fix
```bash
docker compose -f docker-compose.staging.yml up -d --build api web
```

4. Verify staging is really running the new code
```bash
docker compose -f docker-compose.staging.yml ps
docker logs --tail 100 recurvos-staging-api
docker logs --tail 100 recurvos-staging-web
```

5. Re-test the bug in staging
- If the fix touched backend logic only, create a fresh test record after deploy when old staging data may already be polluted
- If the fix touched both backend and frontend, always rebuild both `api web`
- If the app still looks old, confirm `git log --oneline -5` on staging matches local `develop`

Important reminders
-------------------
- Staging server should stay on `develop`
- Production server should stay on `main`
- Merge to `develop` first, test staging, then merge `develop` into `main`
- Localhost does not need `VITE_PUBLIC_SITE_URL` or `VITE_APP_SITE_URL` unless you explicitly want split-domain behavior locally
