Branch rules
------------
- `develop` = staging branch
- `main` = production branch
- Staging server should run `develop`
- Production server should run `main`
- Preferred flow: update `develop` first, test on staging, then merge `develop` into `main`

Core workflow
-------------

1. Do your work on `develop` or merge your feature branch into `develop`
2. Push `develop` to GitHub
3. Deploy staging from `develop`
4. Test staging
5. If staging is good, merge `develop` into `main`
6. Deploy production from `main`

Important rule
- Do not merge `main` into `develop` just to deploy staging
- Staging should deploy whatever is already in `develop`

Step 1: Check your local branch
-------------------------------

```powershell
cd C:\recurvos
git status
git branch --show-current
git log --oneline --decorate -5
```

Expected
- Current branch should be `develop`
- Working tree should be clean before merge/push if possible

Step 2: Put your latest code into develop
-----------------------------------------

If you worked directly on `develop`
```powershell
cd C:\recurvos
git checkout develop
git pull origin develop
git add .
git commit -m "Short clear message"
git push origin develop
```

If you worked on a feature branch
```powershell
cd C:\recurvos
git checkout develop
git pull origin develop
git merge feature/short-description
git push origin develop
```

Verify local `develop`
```powershell
git log --oneline --decorate -5
git log --oneline origin/develop -5
```

Expected
- Your latest commit should appear on `develop`
- `origin/develop` should match after push

Step 3: Deploy develop to staging
---------------------------------

SSH into staging
```bash
ssh server@115.187.22.70 -p 6262
```

Then run
```bash
cd /srv/recurvos/staging/repo
git status
git branch --show-current

git log --oneline -5
git log --oneline origin/develop -5
git checkout develop
```

Expected
- `git branch --show-current` = `develop`
- `git rev-parse HEAD` and `git rev-parse origin/develop` should match after pull
- `api` and `web` should rebuild and restart

Use this when needed

- check `web` logs if frontend build/runtime looks wrong
- check `worker` logs if reminders or WhatsApp-related flows are affected

When staging says already up to date but code looks old
------------------------------------------------------

Check the real remote branch state
```bash
cd /srv/recurvos/staging/repo
git fetch origin
git log --oneline origin/develop -5
git rev-parse HEAD
git rev-parse origin/develop
```

Interpretation
- If `HEAD` matches `origin/develop`, staging already has the latest `develop`
- If the commit you expect is not in `origin/develop`, you did not push it to `develop`
- If a commit exists only on `main`, staging will not get it

When git pull is blocked by local server changes
------------------------------------------------

1. Check what changed
```bash
cd /srv/recurvos/staging/repo
git status
git diff -- docker-compose.staging.yml
```

2. If the local server change must be kept, stash it
```bash
git stash push -m "staging local change" -- docker-compose.staging.yml
```

3. Update the repo
```bash
git fetch origin
git checkout develop
git pull origin develop
```

4. Re-apply the stash only if still needed
```bash
git stash list
git stash pop
```

5. Rebuild the needed containers
```bash
docker compose -f docker-compose.staging.yml up -d --build api web
```

Hotfix workflow
---------------

Use this only for urgent production fixes.

1. Start from `main`
```powershell
cd C:\recurvos
git checkout main
git pull origin main
git checkout -b hotfix/short-description
```

2. Commit and push the hotfix
```powershell
git add .
git commit -m "Hotfix short clear message"
git push -u origin hotfix/short-description
```

3. Merge hotfix into `main`
```powershell
git checkout main
git pull origin main
git merge hotfix/short-description
git push origin main
```

4. Deploy production from `main`
```bash
ssh server@YOUR_PROD_IP -p YOUR_PORT
cd /srv/recurvos/prod/repo
git checkout main
git pull origin main
docker compose -f docker-compose.yml up -d --build api web
```

5. Back-merge production fix into `develop`
```powershell
cd C:\recurvos
git checkout develop
git pull origin develop
git merge main
git push origin develop
```

6. Redeploy staging
```bash
ssh server@115.187.22.70 -p 6262
cd /srv/recurvos/staging/repo
git checkout develop
git fetch origin
git pull origin develop
docker compose -f docker-compose.staging.yml up -d --build api web
```

Important reminders
-------------------
- Staging gets code from `develop` only
- Production gets code from `main` only
- If a commit is only on `main`, staging will not get it
- If a commit is only on `develop`, production will not get it
- `git fetch origin` before comparing logs
- Rebuild `api` whenever backend logic changed
- Rebuild `web` whenever frontend code changed
- Rebuild both `api web` when a feature touches end-to-end behavior
